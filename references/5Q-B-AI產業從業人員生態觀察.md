# 摘要

AI 業存在一條廣為人知的「鄙視鏈」——研究員鄙視工程師，工程師鄙視應用開發者，應用開發者鄙視只會用 API 的人，最底層是「我用 ChatGPT 做的」。這條鏈看似是職業優越感的社會現象，但若以五行框架解讀，其背後是一套完整的**能量生剋結構**：每個職能角色承載一種五行德性，相互之間的張力正是產業系統保持活力的制衡機制。

鄙視，不過是相剋的民間說法。

# 一、五行對應：AI 從業人員的職能映射

## 1.1 對應原則

五行不是標籤，而是**功能性角色的動力描述**。同一個人可能在不同時期或不同情境下體現不同的五行屬性。以下映射描述的是職能在生態系統中的主要動力方向，而非對個人的本質定性。

## 1.2 對應表

| 五行 | 德性 | 職能角色 | 核心功能 | 典型話語 |
|:---:|:---:|:---|:---|:---|
| **木** | 定界 | 研究員、理論建構者 | 劃定問題邊界，開拓可能性空間 | 「這個方向值得研究」 |
| **火** | 顯化 | ML Engineer、訓練工程師 | 將潛在結構燃燒成可見的模型輸出 | 「loss 收斂了，跑起來了」 |
| **土** | 包容 | MLOps、推理/基礎設施工程師 | 承載所有系統，整合各方，讓一切得以運作 | 「我來把它部署好」 |
| **金** | 精煉 | 應用開發者、產品工程師 | 去除雜質，將能力提煉為用戶能使用的形式 | 「這個功能做出來了」 |
| **水** | 深潛 | 用戶、終端使用者 | 消化、積累、反思，將體驗轉化為下一輪需求 | 「這裡不對，那裡有問題」 |

## 1.3 角色說明

**木（研究員）**：木德在於立論域。研究員決定「研究什麼」，這是整個生態最上游的定界動作。沒有木，系統無邊界，火將漫燒無方向。

**火（ML Engineer）**：火德在於顯化。訓練一個模型，是把龐大的潛在結構（資料、架構、參數空間）燃燒成一個具體的、可調用的存在。這是最劇烈、最消耗資源的轉化過程。

**土（MLOps/推理工程）**：土德在於承載與包容。基礎設施工程師是生態中最容易被忽視、也最不可或缺的角色。土崩則萬物無所依，再好的模型也無法抵達用戶。

**金（應用開發）**：金德在於精煉與收斂。應用開發者從模型能力的原礦中，提煉出用戶實際需要的形狀。這是一個去除雜質、賦予形式的過程。

**水（用戶）**：水德在於深潛與回流。用戶不只是消費者，他們是整個循環的淵藪——用戶的反饋、需求、困惑，是下一輪研究方向的水源。水枯則木無所生，整個循環停止。

# 二、相生循環：能量如何流動

## 2.1 正向循環（木生火，火生土，土生金，金生水，水生木）

```
用戶痛點與需求（水）
    ↓ 水生木
研究員發現值得探索的問題（木）
    ↓ 木生火
ML Engineer 訓練出對應的模型能力（火）
    ↓ 火生土
MLOps 將模型部署為穩定服務（土）
    ↓ 土生金
應用開發者打磨出可用產品（金）
    ↓ 金生水
用戶使用、反饋、積累新的認知（水）
    ↓ 循環
```

這是 AI 產業健康運作時的理想能量流。每個角色都從上游獲取能量，並向下游輸出轉化後的成果。

## 2.2 生的條件

相生不是自動的——它需要**介面通暢**：

- 水生木：研究員要真的聽用戶聲音，而不是自己選題
- 木生火：論文要可實作，不只是理論
- 火生土：模型要可部署，不只是 benchmark 好看
- 土生金：基礎設施要開放、可靠，應用開發才能建在上面
- 金生水：產品要真正解決問題，用戶才有有意義的反饋

**介面阻塞是 AI 產業失衡的主要原因之一。**

# 三、相剋張力：鄙視鏈的五行解讀

## 3.1 相剋不是壞事

五行相剋（木剋土，土剋水，水剋火，火剋金，金剋木）是**制衡機制**，不是敵對關係。健康系統中，剋制防止某一行過旺，維持整體均衡。

鄙視鏈，正是相剋在人類社會的民間顯現。

## 3.2 剋制結構解析

| 剋制方向 | 表現形式 | 制衡意義 |
|:---|:---|:---|
| **金剋木**（應用剋研究） | 「你的論文跑不到 production，有什麼用？」 | 防止研究脫離現實，過度抽象 |
| **木剋土**（研究剋 MLOps） | 「你只是工程，沒有創新」 | 防止基礎設施工程師自滿，忽略方向性問題 |
| **土剋水**（MLOps 剋用戶） | 「這個需求我們架構撐不住」 | 防止用戶需求無限膨脹，超出系統承載能力 |
| **水剋火**（用戶剋 ML Engineer） | 「你的模型在真實場景裡根本沒用」 | 防止模型訓練脫離實際使用情境 |
| **火剋金**（ML Engineer 剋應用） | 「你只是 API 調用，算什麼工程師」 | 防止應用層對底層技術的過度簡化與誤用 |

## 3.3 剋制失控：鄙視變成傷害

相剋過度，則成**相乘**（加重傷害）或**相侮**（反剋）：

- **相乘**：ML Engineer 持續打壓應用開發者的自信，導致優秀的應用人才出走，金德枯竭，整個產業的落地能力下降
- **相侮**：用戶集體憤怒（水旺），反過來直接衝擊研究方向（剋木），造成研究資金撤回、方向強制轉向

**鄙視鏈在適度時是制衡，過度時是產業內耗。**

# 四、失衡診斷：各行過旺與過衰的症狀

## 4.1 木旺（研究過剩）

**症狀**：大量論文，少數產品。每年 NeurIPS 湧現數千篇，但真正影響用戶生活的技術比例極低。

**表現**：
- 研究員以 paper 數量衡量一切價值
- 「可重現性危機」（Reproducibility Crisis）
- benchmark 刷分成為目的，而非手段

**制衡**：金剋木——應用開發與市場要求對研究產生更強的反饋壓力

## 4.2 火旺（模型炫技氾濫）

**症狀**：模型能力競賽失控，推理成本高到無法大規模部署。每隔幾週就有「最強模型」，但實際可用的產品沒有跟上。

**表現**：
- 參數規模競賽（size race）
- 推理成本讓中小開發者無法負擔
- 「能力有了，但用不起」

**制衡**：水剋火——用戶的實際付費意願與使用反饋，迫使模型走向輕量化、效率化

## 4.3 土崩（基礎設施失能）

**症狀**：模型好、應用也在，但 serving 不穩定、延遲高、成本失控，整個生態的信任基礎動搖。

**表現**：
- API 頻繁宕機
- 供應商集中風險（少數公司控制關鍵基礎設施）
- 小型應用開發者無法預測成本

**後果**：土崩則金無所附，所有應用層的努力付諸東流。

## 4.4 金缺（產品轉化不足）

**症狀**：技術能力已經夠強，但優秀的應用開發者稀缺，大量技術能力無法轉化為用戶價值。

**表現**：
- 「技術上能做，但沒人做成好產品」
- 應用開發被鄙視，人才不願進入這個職能
- Prompt engineering 被輕視，但實際上是金德的具體表現之一

## 4.5 水枯（用戶聲音消失）

**症狀**：產業進入封閉的技術自嗨循環，失去與真實用戶需求的連結。

**表現**：
- 「AI for AI's sake」——為了展示技術而做功能
- 用戶需求調研被視為次要工作
- 產品功能越來越多，但核心問題沒有解決

**後果**：水枯則木無所生，研究方向失去真實錨點，整個生態進入脫離現實的自循環。

# 五、特殊角色：觀察者位置的重要性

## 5.1 全棧個體（五行兼具）

在 AI 時代，出現了一種新型從業者：借助 AI 工具，一個人可以在不同時期體現不同五行德性——用 AI 幫自己做研究（木）、構建應用（金）、甚至理解底層原理（火）。

這不是「假裝全棧」，而是**借助 AI 工具降低了各德性的切換成本**。五行在一人之中流轉，形成微型的自洽生態。

## 5.2 「用 AI 做的」不在鏈底

傳統鄙視鏈將「用 ChatGPT/Claude 做的」放在最底端。但五行視角提供了不同的解讀：

- 若此人能設計系統、定義問題、整合成果並交付用戶價值——他在流轉木、金、水三德
- 缺少的只是火（自己訓練模型）與土（自己維護基礎設施）
- 但這兩德的功能，已由 AI 服務提供者代為承擔

**鄙視鏈是線性排序的思維，五行是循環結構的思維。後者更接近真實。**

# 六、歸元條件：健康 AI 產業生態的五德均衡

依據 5Q 框架的歸元理論，一個系統在五德均衡、相生流暢、相剋適度時，達到最優運作狀態。

對 AI 產業生態而言，歸元條件如下：

| 五行 | 歸元表現 |
|:---:|:---|
| 木 | 研究方向有真實問題驅動，論文可落地，邊界清晰 |
| 火 | 模型訓練與部署成本合理，能力與效率兼顧 |
| 土 | 基礎設施穩定可靠，開放程度足夠，降低應用層門檻 |
| 金 | 應用開發受到尊重，產品轉化路徑通暢 |
| 水 | 用戶聲音被真正聆聽，反饋回路有效驅動上游 |

**當前 AI 產業的主要失衡**：木旺（研究過剩）+ 火旺（模型競賽）+ 水枯（用戶需求脫節）——典型的「上旺下虛」格局。

---

# 七、結語

鄙視鏈的存在，說明這個產業的生態結構是真實的——有分工、有張力、有動力流動。用五行解讀它，不是要消滅鄙視，而是理解**鄙視背後的結構邏輯**：每一個「鄙視方向」都對應一個相剋關係，而相剋的存在正是系統自我調節的方式。

問題不在於有相剋，而在於剋制是否適度、生的介面是否通暢、水是否真的在滋養木。

> 一個健康的產業生態，不是消滅鄙視鏈，而是讓相剋成為制衡、讓相生成為主旋律。
